# Design Document

> Техническая стратегия достижения цели из requirements.md: полноценная работа
> NVIDIA RTX (Ada AD104) в macOS Tahoe на Hackintosh — либо результат, либо
> доказанная невозможность + испытанный обход (R10).

## Overview

Драйвер — это стек слоёв (L1–L6). Мы строим его снизу вверх, на каждом слое —
**точка проверки на железе** (HW-лог, R11) и **гейт решения**: продолжаем, либо
фиксируем стену доказательством и пробуем обход (R10).

Идём двумя дорожками одновременно:
- **Achieve-track** — реализовать слой и подтвердить на железе.
- **Prove/bypass-track** — если слой упирается в ограничение Apple (а не в наш код),
  собрать техническое доказательство и спроектировать/испытать обход.

**Канал доставки** — **kext, инжектируемый OpenCore**. Это решение вынужденное и
обоснованное: DriverKit/dext **не поддерживает графику** (см. macos-graphics-stack.md),
поэтому dext годится максимум для L1–L4 экспериментов, но дисплей/ускорение (L5–L6)
требуют kernel-резидентного драйвера в модели IOGraphics/IOAccelerator, как у AMD.

**Эталон** — поддерживаемая AMD-карта на той же Tahoe. Все «как у AMD» проверяются
сравнением с реальным поведением AMD на стенде.

## Architecture

### Слои и карта замков Apple

```
L1 PCIe bring-up        — найти карту, BAR0, PMC_BOOT_0            [открыто]
L2 GSP bring-up         — FWSEC-FRTS→WPR2→Booter→GSP-RM→RPC         [открыто, наш код]
L3 Memory management    — GMMU/VRAM через RPC к GSP-RM              [открыто]
L4 Command submission   — каналы/очереди                           [открыто]
---------------------------------------------------------------- линия замков Apple
L5 Display / modeset    — IOFramebuffer + интеграция WindowServer   [ЗАМОК: приватные IO*-интерфейсы]
L6 Acceleration / Metal — IOAccelerator + Metal driver bundle       [ЗАМОК: library validation + закрытый Metal-ABI]
```

L1–L4 — это **портирование известной логики** (open-gpu-kernel-modules / nouveau /
nova-core) в kext. Реверса протокола карты нет — он опубликован NVIDIA.

L5–L6 — это **стена Apple**, не NVIDIA. Здесь Achieve-track неизбежно переходит в
Prove/bypass-track.

### Гейты решения (соответствие R10)

После каждого слоя — гейт:
- **PASS** (есть HW-лог успеха) → следующий слой.
- **BLOCKED** → зафиксировать доказательство (лог/дизасм/эксперимент) → спроектировать
  обход → испытать → классифицировать (постоянный/хрупкий, R9) → решение владельца:
  продолжать обход или зафиксировать как доказанный предел (R10.4).

## Components and Interfaces

### C1 — Identify (L1). Статус: код есть, не на железе.
- `pcie_probe.c` (user-space), `driver/RTXProbe` (kext), `ada_regs.h`.
- Интерфейс: матчинг `10DE:2783`, маппинг BAR0, чтение `PMC_BOOT_0`.
- Закрывает R2. Гейт: HW-лог «вижу Ada AD104».

### C2 — GSP bring-up (L2). Статус: код частично, не на железе.
- `driver/gsp/falcon.*` (reset/DMA-load/boot), `driver/gsp/fwsec_patch.*`,
  `tools/vbios_dump` (FWSEC), `falcon_regs.h`.
- Ещё нужно (PORTING-MAP «не портировано»): kernel DMA-буфер (kext,
  `IOBufferMemoryDescriptor`); `booter.*` (Booter→GSP-RM); `rpc.*` (очереди
  command/status в sysmem, формат RPC).
- Интерфейс к карте: MMIO BAR0 + DMA sysmem; к GSP-RM — RPC.
- Закрывает R3 (частично). Гейт: HW-лог «GSP-RM ответил на первый RPC» + WPR2 set.

### C3 — Memory management (L3).
- VRAM/GMMU через RPC-вызовы к GSP-RM (alloc/map). Источник: open-gpu-kernel-modules
  RM API + nouveau r535.
- Закрывает часть R3. Гейт: HW-лог успешной аллокации/маппинга VRAM.

### C4 — Command submission (L4).
- Каналы/очереди команд (pushbuffer) через RPC. Источник: те же.
- Гейт: HW-лог исполнения тестовой команды GPU (например, copy-engine).

### C5 — Display / modeset (L5). **Замок Apple.**
- Achieve: реализовать `IOFramebuffer`-провайдер на NVDisplay (движок дисплея Ada),
  modeset через RPC к GSP-RM. Открытый референс движка — nouveau `dispnv50`.
- Замок: интеграция с WindowServer и приватные IO*-интерфейсы; на современной macOS
  framebuffer для не-Apple GPU крайне ограничен.
- Prove/bypass: доказать, что именно блокирует (символы/логи WindowServer/IOKit);
  испытать минимальный неускоренный framebuffer и зафиксировать, принимает ли его
  WindowServer.
- Закрывает R5 либо его proof-ветку.

### C6 — Acceleration / Metal (L6). **Главный замок Apple.**
- Achieve: предоставить `IOAccelerator`-сервис + **Metal GPU driver bundle**
  (userspace dylib), который WindowServer/приложения грузят как `MTLDevice`.
- Замки: (а) **library validation** — в WindowServer грузятся только Apple-/
  same-dev-подписанные dylib → сторонний Metal-бандл не загрузится; (б) **Metal
  driver ABI закрыт** — нет публичного интерфейса регистрации GPU.
- Prove/bypass: см. C7. Закрывает R6/R7 либо их proof-ветку.

### C7 — Bypass subsystem (для L5–L6, реализует R10).
- **Library validation:** отключение AMFI через boot-arg `amfi_get_out_of_my_way=1`
  (штатно в экосистеме OpenCore/OCLP) либо Lilu-плагин (AMFIPass-подход), чтобы
  WindowServer мог загрузить наш Metal-бандл. Постоянная настройка OpenCore → по R9
  это «постоянный обход», не хрупкий.
- **Metal-ABI:** реверс приватного интерфейса GPU-бандла Apple под Tahoe
  (дизассемблер AMD-бандла как образца «как выглядит легальный бандл»). Это самая
  тяжёлая и нестабильная часть — кандидат на «доказанный предел», если не выйдет.
- **Подпись/инъекция:** Lilu для патчей в рантайме (по образцу WhateverGreen для AMD).
- Каждый обход документируется: сработал ли, риск, стабильность (R9/R10.3).

### C8 — Verification harness (реализует R11).
- Каждый слой даёт артефакт в `docs/hw-dumps/`: лог `log show`, дамп регистров,
  скрин, замер бенча.
- Сравнение с **AMD-эталоном** на том же стенде (R1.3, R8.2).
- CI остаётся **только compile-check** — не засчитывается как «работает».

## Data Models

Все структуры — в `docs/PORTING-MAP.md` (источник ↔ наш код), сверены с upstream:
- Идентификация: `NV_PMC_BOOT_0` (ada_regs.h).
- Falcon: PFALCON/PFALCON2 регистры (falcon_regs.h), база GSP 0x110000.
- FWSEC: `FalconUCodeDescV2/V3`, PmuLookupTable, DMEMMAPPER/FrtsCmd (fwsec_patch).
- GSP RPC (L2 завершение): command/status message queue —
  open-gpu-kernel-modules `message_queue_priv.h`, nouveau `r535.c` (ещё не портировано).
- L3–L6 модели данных добавляются по мере портирования, с записью в PORTING-MAP.

## Error Handling

- **Гейт-протокол (R10):** на BLOCKED слой обязан произвести: (1) доказательство
  (минимально воспроизводимый эксперимент + лог/символ), (2) дизайн обхода, (3)
  результат испытания обхода, (4) классификацию R9.
- **Стабильность (R4):** невосстановимая ошибка GPU → диагностика в лог, без kernel
  panic всей системы; watchdog на RPC к GSP с таймаутом.
- **Прошивки (R12):** FWSEC/Booter/GSP-RM грузятся как есть; единственный
  санкционированный патч — FWSEC init_cmd + signature (fwsec_patch).

## Correctness Properties

Инварианты, которые ДОЛЖНЫ выполняться на всём протяжении проекта (трассируемы к
требованиям).

### Property 1: HW-доказательность (R11)
**Validates: Requirements 11.1, 11.2, 11.3**
Ни одна способность не помечается «работает» без артефакта с реального стенда.
«Компилируется в CI» и «сверено с исходником» — отдельные, более низкие статусы.

### Property 2: Прошивки неизменны (R12.2)
**Validates: Requirements 12.2**
FWSEC/Booter/GSP-RM загружаются байт-в-байт; единственная допустимая модификация —
патч FWSEC (init_cmd + signature) в `fwsec_patch`. Любая иная модификация прошивки —
нарушение инварианта.

### Property 3: Трассируемость к источнику (R12.1)
**Validates: Requirements 12.1**
Каждый регистр/структура/последовательность имеет запись в `PORTING-MAP.md` со
ссылкой на upstream. Непроверенное помечено `TODO: verify` и не используется как факт.

### Property 4: Гейт-инвариант (R10)
**Validates: Requirements 10.1, 10.2, 10.3, 10.4**
Слой со статусом BLOCKED обязан иметь все четыре артефакта (доказательство, дизайн
обхода, результат испытания, классификация R9) до пометки «доказанный предел».

### Property 5: Повседневная стабильность (R9)
**Validates: Requirements 9.1, 9.2, 9.3**
В продакшн-конфигурации нет обходов, требующих ручного действия на каждую загрузку;
допустимы только постоянные настройки OpenCore.

### Property 6: Изоляция отказа (R4.3)
**Validates: Requirements 4.3**
Ошибка GPU не приводит к kernel panic всей системы — только к диагностике и
деградации драйвера.

## Testing Strategy

- **На железе (основное, R11):** каждый слой — отдельный HW-лог на целевом стенде
  (Tahoe + RTX 4070 Super). Headless-проверка (регистры/mailbox/WPR2) там, где нет
  вывода (до L5).
- **AMD-эталон:** те же сценарии на поддерживаемой AMD-карте → сравнение поведения и
  перформанса (R1.3, R8).
- **CI:** компиляция kext/dext/утилит (есть) — гейт качества кода, НЕ доказательство
  работы.
- **Доказательная база замков:** дизассемблер/символы WindowServer, amfid, Metal-
  бандлов — артефакты в репозиторий как часть proof (R10).

## Phasing & Decision Gates

| Фаза | Слои | Выход (HW-лог) | Гейт |
|---|---|---|---|
| P1 | L1–L2 | GSP-RM отвечает на RPC, WPR2 set | технический фундамент подтверждён |
| P2 | L3–L4 | VRAM alloc + исполнение команды GPU | карта реально работает (compute-уровень) |
| P3 | L5 | картинка на мониторе RTX | **первый замок Apple** — здесь вероятен BLOCKED→proof/bypass |
| P4 | L6 | RTX как MTLDevice, ускорение | **главный замок** — library validation + Metal ABI |
| P5 | — | стабильность 24ч + перф vs AMD | R4, R8, R9 |

Прохождение P2 уже даёт ценный достижимый результат (карта живёт и считает) даже
если P3–P4 упрутся в замок — это и есть честная развилка R10.

## Risks

- **L5–L6 — закрытая модель Apple.** Главный риск проекта. Может закончиться
  доказанным пределом (R10.4b), а не картинкой. Это допустимый исход по требованиям.
- **Подписанные прошивки (Booter/GSP-RM):** взять и грузить — лицензионный вопрос
  (linux-firmware), технически как есть.
- **Совместимость с обновлениями Tahoe:** приватные интерфейсы и Metal-ABI меняются;
  обход по C7 может ломаться (R9 — классифицируем).
- **Нет macOS-стенда с RTX у разработчика сейчас** — на железе ничего не
  проверялось; до появления стенда статусы остаются «компилируется»/«сверено», не
  «работает» (R11).
