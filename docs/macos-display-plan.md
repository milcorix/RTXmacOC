# RTXmacOC — план вывода картинки в macOS (боевой, без «невозможно»)

Цель (в порядке приоритета):
1. **Картинка на мониторе именно в macOS** через нашу RTX (не только Linux/VFIO).
2. **Базовое использование GPU** — хоть какое-то реальное ускорение (2D/blit/скролл/видео),
   позже — акселерация браузера/приложений.
3. Полный Metal/3D — НЕ сейчас.

Принцип: нет готового пути → ищем обход; нет обхода → реверсим закрытое; нет и этого →
пишем своё. Ниже — что именно и в каком порядке.

---

## Ключевой факт (корректировка старого блокера)

`docs/macos-graphics-stack.md` пугает «тремя замками». Уточнение: **Library Validation +
приватный Metal-интерфейс закрывают только УСКОРЯЮЩИЙ путь** (GPU-бандл грузится в
адресное пространство WindowServer/приложений). **НЕускоренный `IOFramebuffer`-драйвер —
отдельный путь и он живой:**
- `VMsvga2` (Zenith432) — сторонний `IOFramebuffer` для VMware SVGA, даёт рабочий стол.
- `VBoxVideo`, QEMU/virtio-gpu FB — то же.
- macOS в VM без passthrough, safe-mode, и загрузка до акселератора — рисуют через простой
  фреймбуфер + софт-композитинг WindowServer.

Значит **пункт 1 (картинка) достижим без обхода подписи**. Пункт 2 (ускорение) — да, там
уже AMFI/LV и реверс `IOAccelerator`, это грязная фаза 2.

## Критический путь

### Фаза 0 (СЕЙЧАС, общая для Linux и macOS): голова физически сканирует
Чтобы показать картинку на ЛЮБОЙ ОС, надо физически поднять голову: PLL пиксельклока +
питание OR-пэдов. На GSP это делает display-супервизор прошивки GSP по core UPDATE.
Наш текущий блокер (голова не сканирует, OUTPUT_RESOURCE/PROCAMP → INVALID_ARG,
GET_ACTIVE timeout) — ровно это. **Код modeset через GSP-RM переносится в macOS-kext 1:1**
(та же прошивка GSP, те же RM-контролы и методы каналов). Поэтому доводим на Linux
(быстрый цикл), потом переносим.

Что реверсить/изучать здесь:
- Почему GSP не доводит modeset: (а) не заармлена GSP event/interrupt подсистема
  (`nvkm_gsp_intr_stall` + `r535_disp_intr` + `nvkm_gsp_device_event_ctor` HOTPLUG/DP_IRQ),
  или (б) выход-методы отвергаются из-за состояния OR, которое обычно выставляет супервизор.
- Полный порядок реального modeset-коммита в nouveau r535 + что ГSP ждёт по прерываниям.

### Фаза 1 (картинка в macOS): kext + IOFramebuffer, БЕЗ ускорения
1. **Порт ядра драйвера в macOS-kext** (`milcorix`): `IOPCIDevice` matching на `10DE:2783`,
   map BAR0/BAR1, тот же GSP-boot (слои 2–4) + modeset (слой 5). Наши модули уже переносимы
   через колбэки `nv_mmio_t` — это и был расчёт.
2. **Subclass `IOFramebuffer`**: отдать WindowServer наш scanout-фреймбуфер во VRAM.
   - `getPixelInformation`/`getDisplayModeCount`/`getDisplayModes`/`getInformationForDisplayMode`
     — из EDID (у нас уже читается).
   - `getApertureRange` → вернуть маппинг нашего FB (VRAM BAR1 или системная память).
   - `setDisplayMode` → наш GSP-modeset на выбранный режим.
   - `enableController`, `connectFlags`, `getAttribute(kConnection*)`.
3. **WindowServer композитит софтом** в наш FB → рабочий стол появляется.

Что реверсить/изучать здесь:
- `IOFramebuffer.h` / `IOGraphicsTypes.h` (публичные, но капризные), поведение на Big Sur+.
- Исходники `VMsvga2` и `WhateverGreen` (acidanthera) — как реальный сторонний FB
  публикует режимы и держит WindowServer на современной macOS; какие атрибуты обязательны.
- `AppleGraphicsDevicePolicy` (`agdpmod`), board-id — чтобы macOS не отфутболил наш GPU.
- Как отдать FB: (а) выделить во VRAM и маппить в BAR1-апертуру, или (б) scanout из
  системной памяти (проще для DMA-когерентности на старте).

### Фаза 2 (базовое ускорение, грязная): реверс IOAccelerator / обход подписи
- **Снять замки загрузки кода**: SIP off, `amfi_get_out_of_my_way=1`, AMFIPass (как OCLP) —
  чтобы наши бандлы грузились в WindowServer/приложения.
- **Минимальный акселератор**: не полный Metal, а точечно — 2D-blit/заливка/скролл через
  наш уже рабочий FIFO/CE (слой 4 умеет исполнять команды GPU!). Экспонировать как
  `IOAccelerator`-заглушку или перехват узких путей композитинга.
- **Видео-декод (NVDEC)** как отдельный «полезный GPU-путь» — часто проще, чем 3D.
- Реверс приватного интерфейса `IOAccelerator`/`IOGPU` (меняется по версиям) — большой RE,
  делаем инкрементально после картинки.

## Порядок работ (что делаем прямо сейчас)

1. **[Фаза 0] Довести головy до scanout на Linux** — точечная диагностика супервизора GSP
   за один прогон (много регистров дисплея/PLL/SOR + event/interrupt арминг), затем фикс.
   Это разблокирует и Linux-картинку, и переносится в kext.
2. [Фаза 1] Каркас macOS-kext + `IOFramebuffer` (можно начинать параллельно: скелет класса,
   matching, маппинг BAR — не требует железа для компиляции).
3. [Фаза 1] Перенос GSP-boot+modeset в kext, первый scanout под WindowServer.
4. [Фаза 2] Снятие замков + минимальный акселератор через FIFO/CE.

Приоритет реверса №1: **GSP display-супервизор** (Фаза 0). №2: **IOFramebuffer на Big Sur+**
(через VMsvga2/WhateverGreen). №3: **IOAccelerator + AMFI/LV** (Фаза 2).
